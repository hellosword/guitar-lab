## 1. 项目概述

**Guitar Lab** 是一款面向吉他爱好者的训练应用系列，核心目标是通过游戏化练习建立"位置-音名-唱名-音程-和弦"的直觉映射。

产品拆分为 **两个独立 App**（非单 App 多模块）：

| App | 名称 | 核心功能 | 技术路线 |
|-----|------|---------|---------|
| **App 1** | 指板记忆游戏 | 指板记忆训练（6 层认知难度） | Vite + React + TS + Tailwind，PWA / Capacitor |
| **App 2** | 节奏工坊（待定） | 高级节拍器 + 节奏打分 | 待定，原生或 Web 视精度需求而定 |

**当前聚焦：App 1（指板记忆游戏）**。App 2 仅做技术预研，不进入开发。

### 核心原则
- **Web-First**：优先 PWA，零安装门槛
- **跨平台**：App 1 通过 Capacitor 打包 iOS/Android
- **MVP 优先**：验证核心用户群后再扩展

## 2. Roadmap

```
Phase 1: 指板记忆游戏（当前）
  ├── 搭建项目骨架（Vite + React + TS + Tailwind）
  ├── 核心组件：SVG 指板图、音名/唱名选择器
  ├── 游戏引擎：题库生成、答题校验、难度曲线
  ├── Level 1~3 玩法实现
  └── PWA 部署 + Capacitor 打包

Phase 2: 评估与迭代（App 1 发布后）
  ├── 收集用户反馈与留存数据
  ├── 解锁 Level 4~6 高阶玩法
  └── 决定是否启动 App 2

Phase 3: 节奏工坊（可选）
  ├── 高精度节拍器（复杂节奏型编辑）
  └── 节奏打分（麦克风采集 + onset 检测）
```

> 双 App 架构的详细分析见 `docs/product/two-app-architecture.md`。

## 5. 代码风格与约定

- **语言**：源代码与注释用**中文**；变量/函数/类型命名用**英文**
- **TS 严格模式**：乐理数据结构复杂，禁止 `any`
- **状态管理**：所有游戏状态变更通过 Zustand store，禁止直接修改状态对象
- **指板渲染**：唯一方案为 **SVG**（不可改为 Canvas）
- **交互规范**：以 `docs/product/guitar-fretboard-game-design.md` 第 4 节"答题交互规范"为准

## 7. 题库系统架构原则（核心指导方针）

题库系统是 App 1 的核心引擎，其架构必须满足以下**不可妥协**的原则：

### 7.1 可拓展性优先

- 必须非常容易支持**新的题目类型**（如未来新增 Level 6 的多声部题型）
- 必须非常容易支持**新的题目形式**（如判断对错、单选、多位置点击、顺序点击等）
- 必须非常容易支持**新的具体题目**（无需修改代码即可增加新关卡或新题）
- **禁止**用硬编码的 `if/else` 或 `switch` 分支来处理不同题型

### 7.2 数据驱动 / 配置驱动

- 所有题目必须通过**参数化配置**定义，而非代码逻辑写死
- 题目 = 配置数据 + 通用渲染/校验引擎
- 配置应使用人类可读的格式（JSON / YAML），便于：
  - 非程序员（如音乐老师）参与设计题目
  - 用户自定义题库的热加载
  - 题库的跨版本兼容与迁移

### 7.3 架构实现方向

```
题目配置（JSON/YAML）
      ↓
题目解析器（验证 schema，加载配置）
      ↓
游戏引擎（根据配置生成题目实例）
      ↓
渲染层（根据题目类型选择对应的 UI 组件）
      ↓
校验器（根据配置的答案规则判定正误）
```

- **题目类型注册表**：每种新题型只需"注册"到系统中，无需修改现有代码
- **答案类型与交互组件解耦**：答案类型（音名/位置/音程/唱名）决定交互组件的选择，而非题型硬编码
- **难度曲线也是配置**：Level 1~6 的解锁条件、范围限制、混合策略均为可配置数据

> 详细设计见 `docs/tech/question-bank-architecture.md`。

## 10. Next Steps 管理规则

### 10.1 目的

项目并行事项较多，为防止 next step 散落在各处对话中，统一记录到 `docs/next.md`。

### 10.2 AI 行为规则

- 当 AI 在对话中提出**合理的下一步行动建议**（如"接下来你可以做 X"、"建议先完成 Y"），应主动将该事项追加到 `docs/next.md` 的对应优先级区域。
- 追加前检查：该事项是否已存在于 `next.md` 中，避免重复。
- 追加格式遵循 `next.md` 中的模板（来源、内容、预期产出）。
- 如果用户说"这个以后再说"，则将事项移到 `next.md` 的"低优先级"或"待评估"区域。
- 如果用户完成了某项，AI 应将其从"待办"区域移到"已完成"区域，并标注日期与版本号。

### 10.3 版本记录规则

- 可见产品改动完成时，必须递增 `src/appVersion.ts` 中的页面版本号。
- 同步更新 `package.json` 与 `package-lock.json` 中的版本号。
- `docs/next.md` 的"已完成"表必须包含"版本/标签"列，新增完成项必须写入对应版本号。
- AI 在最终回复中应明确告诉用户当前版本号，便于用户刷新页面确认。
- 代码提交信息建议带版本号，格式：`v0.0.2: Add visible app version`。
- Git tag 建议与页面版本号一致，格式：`v0.0.2`。

### 10.4 灵感记录规则

- 用户提到的**非正式想法、假设、灵感**（即使不打算立即实现），应记录到 `docs/inbox.md`。
- `inbox.md` 是"防止遗忘"的收集箱，不要求结构化，按日期分组追加即可。

### 10.5 文档位置

| 文档 | 用途 |
|------|------|
| `docs/next.md` | 正式下一步行动清单，有优先级、有预期产出 |
| `docs/inbox.md` | 灵感与想法收集箱，非正式、不承诺实现 |
