## 2026-04-25

### 智能出题 vs 手动选关

用户不应该手动选择关卡。用户只需要表明他要练习 G 大调，系统根据他的历史数据和当前掌握情况，自动为他出题。

**为什么有这个想法**：
- 关卡选择对吉他学习者来说是一种认知负担（"我该练哪一关？"）
- 用户真正关心的是"我想练什么内容"，不是"我要打通哪一关"
- 如果系统能自动判断用户弱项（比如 F# 总是答错），就可以自动增加 F# 相关题目的权重

**待思考**：
- 用户是否需要某种方式表达"我今天想轻松一点"或"我今天想挑战自己"？
- 自动出题的策略算法如何设计？
- 与现有"关卡配置"体系是否兼容？

### 混合训练模式

指板记忆到后期，一种类型一种类型分别训练可能不太好。应该有一种混合模式，把多种题型混合在一起练习。

**为什么有这个想法**：
- 真实演奏中，吉他手需要在音名、唱名、音程、和弦内音之间**即时切换思维框架**
- 分别训练每种映射是奠基，但混合训练才是"学以致用"
- 混合模式可以模拟真实场景："这个音是什么？它在这个调里唱什么？它和根音是什么关系？"

**待思考**：
- 混合模式的解锁时机？（掌握所有基础映射之后？）
- 混合的比例如何控制？
- 是否需要在 UI 上有明确的"当前模式"指示？

## 2026-05-01

### Agent 自循环测试

用户认为 Agent 自循环测试是项目开发中的重要环节。

**为什么有这个想法**：
- 代码生成后，如果 Agent 不能自己构建、启动、访问和验证页面，就会把大量验收负担转移给用户
- 前端项目尤其需要至少覆盖“构建是否成功、页面是否可访问、关键资源是否加载”
- 后续最好能升级到浏览器截图、交互点击、视觉回归和核心流程自动化

**当前处理**：
- 已新增 `npm run test:smoke`
- 该脚本用纯 Node 静态服务器托管 `dist`，检查首页、JS/CSS 资源和 PWA manifest
- 这条链路绕开了 Windows 沙箱中 Vite/esbuild dev server 的 `spawn EPERM` 限制

**待思考**：
- 是否引入 Playwright 做真实浏览器交互测试？
- Codex 桌面环境是否能稳定提供浏览器自动化工具？
- 是否需要在每次实现 UI 后固定执行 `npm run build` + `npm run test:smoke`？

## 2026-05-02

### 六线谱位置到唱名的训练路径

用户在思考指弹练习中，六线谱阅读实际包含旋律、和弦根音与伴奏三部分；其中旋律最好能哼唱出来，因此“六线谱位置 -> 唱名”的映射是核心能力。

**当前假设**：
- 通用训练路径可以先练“六线谱位置 -> 音名”，再练“G 大调下音名 -> 唱名”
- 前一段映射具有跨调通用性，后续在其他调式中也复用
- 但用户担心两步映射在真实演奏中会慢一拍，不如直接训练“G 大调下六线谱位置 -> 唱名”

**待思考**：
- 是否应把“位置 -> 音名”和“位置 -> 调内唱名”设计成两个基础模块，再用混合训练把两者压缩成自动化反应？
- 对初学者而言，直接练调内唱名是否更贴近旋律感，但可能牺牲跨调迁移？
- 游戏中是否需要支持“固定调唱名”和“首调唱名”两种训练视角？

## 2026-05-04

### 自录吉他采样与和弦声音展示

用户提到未来会有和弦声音展示，并且自己有电吉他，可以考虑自录音。

**当前想法**：
- 单音提示、和弦展示、未来节奏训练最好共用一套音频资产路线，而不是临时换采样库。
- 如果目标是“干净、低延迟、可控、可商用/可分发”，自录 DI 干声或近距离干净录音可能比继续寻找免费库更稳。
- 和弦展示不一定需要录下所有和弦，可以先录单音采样，用 Web Audio 同时触发多个音来合成和弦；以后再补真实和弦扫弦/分解和弦采样。

**待思考**：
- 目标音色是干净电吉他、木吉他，还是两套都支持？
- MVP 阶段和弦只需要“同时响起的音高关系”，还是需要真实吉他扫弦手感？
- 自录采样是否按弦/品录，还是按音高多采样录？

## 2026-05-03

### 木吉他音色与更自然的延音

当前练习音频过于短促，听感不像真实吉他；音色也更像简单合成波形，而不是木吉他。后续可以调研或实现真正的木吉他采样音色，并让音符有更绵长、自然衰减的延音。

**为什么有这个想法**：
- 练习目标不仅是点对位置，还包括建立音高记忆；音色越接近真实吉他，迁移到实际弹奏时越自然。
- 过短的声音不利于听清音高，也容易和下一题提示音混在一起。
- 如果未来要做更强的听觉训练，音色和包络会成为核心体验的一部分。

**待思考**：
- 使用真实木吉他采样，还是先优化当前 WebAudio 合成器的包络？
- 每个音是否需要同弦/同品位的不同采样，还是按音高复用采样即可？
- 重播、点击发音、自动提示音是否需要不同音量或包络，避免练习时听感疲劳？

### 六线谱阅读的三层拆解能力

用户提出：对着六线谱弹吉他，尤其是指弹，可能同时包含旋律、和弦/根音、伴奏三个部分。终极能力不是单个位置识别，而是看到一小节六线谱时，能在脑内把音符按功能拆开：哪些是旋律，哪些是和弦或根音，哪些是伴奏织体。

**当前假设**：
- 旋律音应优先映射到唱名，因为它关系到能否哼唱、记忆和理解旋律走向
- 和弦/根音应映射到音名、级数与音程关系，例如根音、三音、五音、七音以及大三度/小三度等结构
- 伴奏不一定主要映射到单个音高名称，而可能更适合映射到节奏型、分解和弦型、低音-高音声部关系、和声填充功能

**待思考**：
- App 是否需要从单音映射训练逐步扩展到“小节内声部分层识别”？
- 伴奏训练的目标应定义为音高识别、节奏型识别、和声功能识别，还是三者组合？
- 对初学者而言，如何把这个终极能力拆成不压垮人的微技能阶梯？
