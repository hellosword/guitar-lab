# Obsidian 技术架构对 Guitar Lab 的借鉴分析

> 分析日期：2026-04-25
> 对应调研文档：`obsidian-architecture.md`
> 目标读者：Guitar Lab 项目技术决策者

---

## 1. 可直接采纳的设计决策

### 1.1 Capacitor 移动端的可行性已获验证 ✅

**Obsidian 的实践**：Obsidian 移动端基于 Capacitor 构建，用户反馈其功能丰富度"几乎与桌面端一致"，包括复杂的图谱渲染、插件加载、文件系统操作等。

**Guitar Lab 的决策验证**：

- 消除了"Capacitor 能否承载复杂游戏交互"的核心顾虑
- 指板图的多点点击、SVG 渲染、音频播放、本地存储等需求在 Capacitor 中完全可行
- **无需调整技术栈**，维持 Vite + React + TS + Capacitor 的方案

### 1.2 Local-first 数据哲学 ✅

**Obsidian 的实践**：所有数据以纯文本 Markdown 存放本地，用户完全拥有数据，无需导出，任何工具都能读取。这成为 Obsidian 的核心竞争优势之一。

**Guitar Lab 的应用**：

| 数据类型 | 存储方案 | 设计原则 |
|---------|---------|---------|
| 用户练习进度 | IndexedDB（Web）/ Capacitor Preferences（App） | 完全本地，不上传 |
| 历史成绩 | IndexedDB | 可导出为 JSON，可备份 |
| 自定义题库 | 本地 JSON/YAML 文件 | 开放格式，可编辑可分享 |
| 应用配置 | 本地存储 | 随设备独立 |

**关键设计**：在 App 内提供明确的"导出我的数据"功能，让用户感知到数据属于自己，增强信任感。

### 1.3 Restricted Mode 的安全设计思路 ✅

**Obsidian 的实践**：默认开启 Restricted Mode（安全模式），第三方代码完全禁止执行。用户必须**主动、明确地关闭**才能安装社区插件。

**Guitar Lab 的应用**：

- 如果未来开放"自定义题库分享"或"社区关卡"功能，默认禁用外部内容执行
- 用户上传/导入的第三方内容需经过明确的用户确认步骤
- 所有涉及代码执行的功能（如用户自定义 JavaScript 逻辑）默认关闭

### 1.4 配置与数据分离 ✅

**Obsidian 的实践**：Vault 级配置存放在 `.obsidian/` 目录，用户数据（笔记文件）与配置分离，便于迁移和版本控制。

**Guitar Lab 的应用**：

```
GuitarLab/
├── config/              # 应用配置（主题、难度偏好、音频设置）
│   ├── settings.json
│   └── theme.json
├── data/                # 用户数据（进度、成绩、解锁状态）
│   ├── progress.db      # IndexedDB 导出
│   └── achievements.json
└── custom/              # 用户自定义内容（题库、节奏型配置）
    ├── custom-levels/
    └── rhythm-patterns/
```

## 2. 需要规避的陷阱

### 2.1 不要引入 Electron ❌

**Obsidian 的代价**：桌面端使用 Electron，安装包 150MB+，内存占用高，被用户诟病"电脑上跑了一堆浏览器"。

**Guitar Lab 的对策**：

- 桌面端使用 **PWA** 即可满足需求
- Guitar Lab 没有深度文件系统集成需求（不需要像 Obsidian 那样频繁读写大量 Markdown 文件）
- PWA 可通过 `vite-plugin-pwa` 快速实现离线可用、可安装到桌面
- 只有在未来需要后台音频（锁屏播放节拍器）时，才考虑原生方案

### 2.2 不要过早设计插件系统 ❌

**Obsidian 的代价**：

- 社区插件无法沙箱化，安全隐患显著
- 需要维护稳定的 Plugin API，向后兼容性负担重
- 插件质量参差不齐，影响产品口碑
- 大量用户反馈"插件装多了移动端崩溃"

**Guitar Lab 的对策**：

- **MVP 阶段不做插件系统**
- 如果后期需要扩展性，优先考虑：
  1. **数据驱动配置**：用 JSON/YAML 定义关卡、节奏型，无需代码
  2. **严格沙箱**：如果必须执行用户代码，使用 Web Worker 隔离
  3. **官方 curated 内容**：由官方审核的关卡包，而非完全开放的插件市场

### 2.3 不要低估同步的复杂性 ❌

**Obsidian 的代价**：iCloud 同步问题是最热门的论坛话题之一，用户频繁遭遇文件丢失、冲突覆盖、索引延迟。

**Guitar Lab 的对策**：

- **App 1（指板记忆游戏）完全单机化**：无需同步，降低复杂度
- 用户进度通过**本地导出/导入 JSON** 实现跨设备迁移
- 如果未来需要云端同步，使用成熟的 BaaS（如 Supabase / Firebase），而非自建同步协议
- 节奏打分功能（App 2）如果涉及数据同步，在 Phase 3 再评估

## 3. 架构层面的启发

### 3.1 文件系统即数据源的"透明性"设计

Obsidian 将 Vault 直接映射到文件夹，任何外部工具都能操作。这种"数据透明"理念可以应用于 Guitar Lab：

| Obsidian | Guitar Lab |
|----------|-----------|
| `.md` 文件即笔记 | JSON 文件即关卡配置 |
| 任何编辑器可读 | 任何文本编辑器可修改题库 |
| Git 可直接版本控制 | 用户可备份/分享关卡文件 |
| 无数据库锁定 | 无专有格式锁定 |

**落地建议**：

- 自定义题库使用人类可读的 JSON 格式，包含完整注释
- 提供"题库格式规范文档"，让高级用户可以自己编写关卡
- 导出功能不仅用于备份，也用于"分享我的自定义关卡给好友"

### 3.2 避免双渲染管线的复杂度

Obsidian 最大的技术债务之一是同时维护 Reading 和 Live Preview 两套渲染管线，插件开发者经常需要为两种模式写两套代码。

**Guitar Lab 的启示**：

- 指板图的"题目展示"和"用户输入点击"使用**同一套 SVG 组件**
- 不要为"展示模式"和"交互模式"维护两套渲染逻辑
- 通过状态（State）区分是"只读展示"还是"可点击输入"，而非两套组件

```typescript
// 推荐：同一组件，通过状态控制交互
<Fretboard
  highlightedPositions={question.positions}  // 题目高亮
  clickable={mode === 'input'}               // 是否可点击
  onPositionClick={handleAnswer}             // 答题回调
/>
```

### 3.3 核心 API 的分层抽象

Obsidian 的 Plugin API 围绕 `Vault`、`Workspace`、`MetadataCache` 三个核心抽象分层，使插件开发直观且一致。

**Guitar Lab 可预留的抽象层**（即使初期不开放插件）：

```typescript
// lib/ 目录下的核心抽象
lib/
├── theory.ts          # 乐理计算引擎（对应 Obsidian 的 Vault）
├── audio.ts           # 音频引擎封装（对应 Workspace）
├── storage.ts         # 数据持久化（对应 MetadataCache）
└── game-engine.ts     # 游戏逻辑编排
```

**设计原则**：

- 每个模块职责单一，接口稳定
- 即使不开放给第三方，内部模块间也通过明确接口通信
- 为未来可能的扩展点预留插槽

## 4. 产品策略层面的借鉴

### 4.1 "默认简洁，可扩展"的产品哲学

Obsidian 的核心体验极其简洁（就是一个 Markdown 编辑器），但可以通过插件变成任务管理器、日历、看板、数据库等任意形态。

**Guitar Lab 的应用**：

- **App 1 的核心体验**：打开即练，3 秒内进入题目
- 高级功能（自定义题库、深度统计、社交分享）不打扰核心流程
- 通过渐进式披露（Progressive Disclosure）引导用户发现高阶功能

### 4.2 社区生态的冷启动策略

Obsidian 的社区插件生态是如何从 0 增长到 2700+ 的？

| 阶段 | 策略 |
|------|------|
| 初期 | 官方提供完善的开发者文档和示例插件 |
| 成长期 | 降低发布门槛：GitHub repo + manifest.json 即可提交 |
| 成熟期 | 官方插件浏览器内置，一键安装 |

**Guitar Lab 的启示**：

- MVP 阶段不需要插件系统，但需要**清晰的内部架构**
- 如果未来开放自定义内容，先从"数据文件（JSON）"开始，而非"代码插件"
- 社区生态的前提是核心产品本身足够好用

### 4.3 平台差异化的克制

Obsidian 移动端虽然功能接近桌面端，但官方明确限制了部分功能（如导出 PDF 在移动端不可用）。

**Guitar Lab 的应用**：

| 功能 | 桌面/PWA | 移动端 App | 差异原因 |
|------|---------|-----------|---------|
| 指板记忆游戏 | ✅ | ✅ | 核心功能，全平台一致 |
| 节奏打分 | ✅ Web Audio | ⚠️ 精度受限 | 移动端音频延迟高 |
| 后台节拍器 | ❌ | 需原生扩展 | PWA 无法后台播放音频 |
| 数据导出 | ✅ | ✅ | 全平台一致 |

**原则**：不为追求"功能一致"而牺牲体验，明确告知用户平台差异。

## 5. 具体行动建议

### 近期（App 1 MVP）

- [ ] 确认 Capacitor 技术路线不变，参考 Obsidian 移动端的实现信心
- [ ] 设计用户数据的本地存储结构，遵循"开放格式、用户拥有"原则
- [ ] 实现"导出/导入我的数据"功能，增强用户信任
- [ ] 在 `lib/` 目录建立清晰的分层抽象，预留扩展点

### 中期（App 1 迭代）

- [ ] 评估是否引入 JSON/YAML 自定义题库功能（数据驱动扩展，非代码插件）
- [ ] 设计 Restricted Mode 类似的安全开关，控制外部内容执行
- [ ] 监测 Capacitor 移动端性能，验证 SVG 指板图在低端设备上的表现

### 远期（App 2 评估）

- [ ] 研究 Obsidian 桌面/移动的功能差异策略，制定 App 2 的平台能力矩阵
- [ ] 如果节奏打分需要高精度，参考 Obsidian "Pro 功能差异化"思路，明确告知用户移动端精度限制

## 6. 总结

| 维度 | Obsidian 实践 | Guitar Lab 建议 |
|------|--------------|----------------|
| 跨平台 | Electron(桌) + Capacitor(移) | PWA(桌) + Capacitor(移) |
| 数据存储 | Local-first Markdown | Local-first JSON/IndexedDB |
| 扩展性 | 代码插件（2700+生态） | MVP不做；后期数据驱动扩展 |
| 安全 | Restricted Mode默认开启 | 借鉴：默认保守、用户主动开放 |
| 同步 | 官方Sync + 第三方兼容 | App1完全单机，后期评估BaaS |
| 渲染 | 双管线（Reading + Live Preview） | 单管线：状态驱动同一SVG组件 |
| 产品哲学 | 默认简洁，可无限扩展 | 打开即练，高级功能渐进披露 |

**核心结论**：Obsidian 的架构选择验证了 Guitar Lab 当前技术路线（Capacitor + Local-first + Web 技术栈）的正确性，同时也提供了关于"何时不做"（Electron、插件系统、复杂同步）的宝贵反面教材。
