# 移动端 Adaptive Shell 交互规格

> 日期：2026-05-04  
> 状态：规格草案，待进入实现  
> 范围：Guitar Lab 的 PC Web、移动端 Web/PWA、未来 Capacitor App 的交互壳层划分。  
> 目标：避免把 PC 页面简单缩小成移动页面，建立“共享训练核心 + 分端交互壳”的长期架构。

---

## 1. 背景

Guitar Lab 当前已经形成比较清晰的训练模型：

- 位置、音名、唱名、六线谱构成认知节点。
- 练习模式是节点之间的认知通路。
- 历史记录、自适应出题、弱点地图围绕通路与位置持续积累数据。
- 指板速查承担全局参考手册角色。

PC Web 第一版已经能承载这些信息，因为大屏适合高密度并排布局：顶部工具栏、练习模式区、题面、答案区、右侧详情、图选弹窗和弱点地图可以同时存在。

但移动端不是“屏幕更窄的 PC”。手机竖屏的核心矛盾不是宽度不够，而是：

- 用户通常单手触控，操作目标需要更大。
- 用户一次只能稳定关注一个任务。
- 多排按钮、说明、详情和图形同时出现会挤掉真正的练习区域。
- 复杂设置更适合临时打开，而不是常驻压在首屏。
- 弱点地图、速查和练习虽然共享数据，但在移动端应成为不同任务场景。

因此，移动端需要独立的交互壳，而不是继续对 PC 页面做局部响应式压缩。

## 2. 调研结论

### 2.1 主流共识

跨平台产品的主流设计不是“一套页面缩放到所有设备”，而是：

```text
共享核心能力
      +
按设备和输入方式选择不同交互壳
      +
组件内部做响应式适配
```

这里可以称为：

```text
Adaptive Shell Architecture
自适应交互壳架构
```

它包含两层含义：

- Responsive：让内容装得下当前空间。
- Adaptive：让界面在当前空间和输入方式下真的好用。

Flutter 官方文档对两者的区分很适合作为本项目原则：responsive 关注 UI 如何适应空间，adaptive 关注 UI 在该空间内是否可用。

### 2.2 典型设计模式

| 来源 | 可借鉴结论 |
|------|------------|
| Flutter adaptive/responsive | 单一代码库可以服务多平台，但布局和输入方式需要同时适配。 |
| Android / Material adaptive layout | 按窗口尺寸切换导航结构，小屏、平板、大屏使用不同布局框架。 |
| Ionic split pane | Web + App Store 发布场景中，大屏显示多视图，小屏折叠菜单，是跨平台壳层分化的典型模式。 |
| Apple HIG | Tab Bar 适合移动端顶层区域切换，Sidebar 需要较多空间，iPhone 上不宜作为主要结构。 |
| React Native platform-specific code | 尽量复用代码，但当平台视觉或行为差异明显时，应拆出平台特定组件。 |
| Slack / Notion | 桌面端侧边栏和高密度导航，移动端使用底部导航或单屏任务流，信息结构相同但交互入口不同。 |

参考资料：

- [Flutter Adaptive and Responsive Design](https://docs.flutter.cn/ui/adaptive-responsive)
- [Android Adaptive Material Guidance](https://developer.android.com/codelabs/adaptive-material-guidance)
- [Ionic Split Pane](https://ionicframework.jp/docs/v7/api/split-pane)
- [Apple Tab Bars](https://developer.apple.com/design/human-interface-guidelines/tab-bars)
- [Apple Sidebars](https://developer.apple.com/design/human-interface-guidelines/sidebars)
- [React Native Platform-Specific Code](https://reactnative.dev/docs/platform-specific-code)
- [Slack Sidebar](https://slack.com/hc/en-us/articles/212596808-Manage-your-sidebar-display)
- [Slack Desktop Navigation Update](https://slack.com/intl/en-gb/help/articles/16764236868755-An-overview-of-Slacks-new-design)
- [Notion Mobile Workspace](https://www.notion.com/zh-tw/help/workspaces-on-mobile)

## 3. 核心原则

### 3.1 共享训练核心，不共享完整页面

必须共享：

- 乐理定义
- 调性规则
- 唱名显示模式
- 指板位置模型
- 题目生成
- 答题校验
- 历史记录
- 自适应出题
- 弱点统计
- 音频播放接口
- 数据导入导出
- 版本与构建标识

不强求共享：

- 页面布局
- 顶部导航
- 模式选择器形态
- 弱点地图呈现方式
- 详情区位置
- 速查入口
- 图选入口
- 移动端手势和底部操作区

### 3.2 PC 是工作台，移动端是练习器

PC Web 的关键词：

- 高信息密度
- 并排比较
- 快速切换
- 右侧详情常驻
- 图选与弱点地图可视化
- 适合调试和深度分析

移动端的关键词：

- 当前任务聚焦
- 单手可点
- 设置后置
- 反馈强
- 少滚动
- 少同时展示
- 随时打开就能练

### 3.3 移动端默认进入 Focus Mode

移动端打开练习页时，不应该先看到大量模式按钮和全局设置，而应该首先看到：

- 当前练习是什么。
- 当前题目是什么。
- 我应该在哪里作答。
- 我答完之后发生了什么。

模式切换、调性、唱名显示、音色、版本信息、图选通路都应通过 Sheet、Drawer 或次级页面打开。

## 4. 架构边界

### 4.1 推荐代码分层

长期建议目录结构：

```text
src/domain/
  乐理、题目、校验、自适应、弱点统计等纯业务核心

src/app/
  全局配置、版本、持久化、应用级状态适配

src/components/
  Fretboard、按钮、选择器、弱点热度图等可复用组件

src/desktop/
  PC Web 页面组合、桌面导航、桌面布局、桌面图选弹窗

src/mobile/
  移动端 Focus Mode、底部 Sheet、移动端导航、移动端弱点页
```

当前不要求一次性重构到上述目录，但新增移动端能力时应逐步向这个方向靠拢。

### 4.2 短期可接受实现

MVP 阶段可以先在现有 `src/App.tsx` 中做壳层拆分：

```tsx
{isMobileShell ? (
  <MobilePracticeShell ... />
) : (
  <DesktopPracticeShell ... />
)}
```

但要求：

- 两个 Shell 只负责组合 UI，不复制题目逻辑。
- 题目状态、答题动作、切题规则和历史记录继续复用同一套接口。
- 公共组件可以根据 `variant` 做小差异，但不要让一个组件内部塞满端侧判断。
- 当某个端侧差异已经影响布局结构，应拆成独立组件，而不是继续堆条件 class。

## 5. 信息架构

### 5.1 PC Web 壳层

PC Web 保持当前方向：

```text
顶部紧凑工具栏
  - 产品名 / 版本 / 构建标识
  - 练习 / 速查
  - 调性
  - 唱名显示
  - 音色

练习模式区
  - 训练组
  - 方向
  - 图选入口
  - 练习 / 查看弱点 tab

主体区
  - 左侧题目与作答
  - 右侧详情、播放、反馈、统计
```

PC 的目标是让开发者和高频用户快速切换、观察、分析。

### 5.2 移动端壳层

移动端建议拆成三个顶层场景：

```text
练习
速查
弱点
```

移动端底部导航第一版可以是：

```text
[练习] [速查] [弱点]
```

也可以先不做常驻底部导航，而是使用顶部轻量入口。但长期看，练习、速查、弱点是移动端用户最容易理解的三个任务场景。

移动端练习页结构：

```text
顶部上下文栏
  - 当前练习名
  - 当前调性
  - 设置按钮

题目区
  - 第 n / 20 题
  - 题目提示
  - 当前目标

作答区
  - 单选按钮组 / 指板 / 六线谱输入

底部动作区
  - 播放音高
  - 下一题
  - 反馈状态
```

模式选择、图选、调性、唱名显示、音色进入 Sheet：

```text
练习设置 Sheet
  - 训练组
  - 方向
  - 当前调性
  - 唱名显示
  - 音色
  - 图选通路
  - 版本 / 构建标识
```

### 5.3 移动端弱点页

移动端弱点页不应照搬 PC 右侧详情或大图布局。

建议结构：

```text
顶部：当前通路 + 调性
主体：弱点热度图 / 弱点列表
底部：开始针对复练
```

第一版可以先支持当前已有弱点地图最成熟的通路，未有数据的通路显示“暂无足够数据”。

## 6. 关键交互规则

### 6.1 练习优先

移动端首屏必须尽量让用户看到题目和作答区。

不应在首屏常驻：

- 多排模式按钮
- 大段说明
- Graph 图
- 版本详情
- 完整弱点列表
- PC 风格右侧详情

### 6.2 设置后置

设置不是消失，而是放入明确入口：

- 当前练习名旁的设置按钮。
- 或底部导航中的“练习”页内 Sheet。

设置打开后可以展示完整配置，但关闭后必须回到干净练习界面。

### 6.3 图选是理解工具，不是移动端日常入口

Graph 对理解认知通路很有价值，但不适合作为移动端常驻日常控件。

移动端策略：

- 日常切换使用分组列表或分段选择。
- Graph 放在“图选通路”入口中。
- Graph 可以横屏或内部横向滚动，不要求塞进竖屏首屏。

### 6.4 详情区移动到反馈层

PC 右侧详情常驻是合理的，移动端不应常驻。

移动端详情可以变成：

- 答题后展开的反馈卡片。
- 点击“详情”后的底部 Sheet。
- 总结页中的本题/本轮分析。

### 6.5 指板是主输入对象

移动端指板题中，指板要优先获得空间：

- 允许卡片内横向滚动。
- 保持足够触控宽度。
- 点击热区按最近弦品判定。
- 反馈标记必须清晰，不能过小。
- 播放音高应与点击自然绑定。

## 7. 第一阶段实现范围

第一阶段目标：让移动端从“PC 页面缩小版”变成“能日常练习的 Focus Mode”。

必须完成：

- 移动端隐藏 PC 顶部多排工具栏，替换为轻量上下文栏。
- 移动端隐藏常驻练习模式大区，替换为“练习设置”Sheet。
- 移动端练习页首屏优先展示题目与作答区。
- 音名定位练习在移动端拥有更清晰的题目、目标音名、指板和反馈布局。
- 右侧详情在移动端移动到题后反馈或折叠详情。
- 设置 Sheet 中可以切换训练组、方向、调性、唱名显示和音色。
- 页面仍显示正式版本与构建标识，但放入设置或较弱位置。

暂不完成：

- 全部页面彻底目录重构。
- Capacitor 原生壳。
- 移动端独立动画系统。
- 所有弱点地图的移动端专属可视化。
- 手势教学和新手引导。

## 8. 后续阶段

### 8.1 第二阶段：移动端弱点与复练

- 弱点页成为移动端顶层场景。
- 每条练习通路显示对应弱点概览。
- 支持从弱点页直接开始针对复练。
- 弱点地图在移动端采用“热度图 + Top 列表 + 点击详情”的结构。

### 8.2 第三阶段：移动端速查

- 指板速查针对触控重新设计。
- 音名/唱名映射面板改为可横向扫视或分段切换。
- 支持长按或点击指板发音。
- 支持全局调性和唱名显示切换。

### 8.3 第四阶段：PWA 与 Capacitor 发布

- 检查 PWA 缓存更新与版本显示。
- 验证手机浏览器、安装到桌面、Capacitor WebView 的差异。
- 补充移动端回归测试和真实设备测试清单。

## 9. 验证标准

移动端打开后，用户应能在 3 秒内回答：

- 我现在练的是什么？
- 当前是什么调？
- 这一题要我做什么？
- 我应该点哪里作答？
- 我答完之后是否正确？

移动端不要求用户在首屏回答：

- 系统支持多少种练习模式？
- 完整认知通路图是什么？
- 当前构建号是什么？
- 所有弱点排行是什么？

E2E 和人工验证重点：

- 手机宽度下无页面级横向溢出。
- 首屏能看到题目和主要作答区域。
- 设置 Sheet 可打开、关闭、切换模式。
- 切换设置后练习能按现有规则重开或更新显示。
- PC 端现有布局和测试不被破坏。
- 指板题在移动端点击、发音、反馈清晰可靠。

## 10. 设计判定原则

后续遇到 PC 和移动端设计冲突时，按以下规则判断：

- 如果差异只是尺寸、间距、字体、换行，用响应式样式解决。
- 如果差异影响信息层级，用端侧布局解决。
- 如果差异影响任务流程，用独立 Shell 解决。
- 如果差异影响业务规则，先检查是否误把交互问题放进了核心逻辑。

最终目标不是维护两套产品，而是维护一套训练系统，以及两套符合使用场景的交互入口。
