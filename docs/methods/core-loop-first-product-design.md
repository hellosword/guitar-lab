# 核心闭环优先的产品设计方法

> 日期：2026-05-01  
> 适用场景：游戏、学习工具、训练工具、AI 辅助产品设计  
> 来源：Guitar Lab MVP 讨论。用户是游戏开发者，该方法与“先做核心玩法，再搭建周边系统”的游戏开发经验一致。

---

## 1. 核心观点

产品设计早期最重要的不是把系统设计完整，而是找出用户真正需要被改变的那个行为闭环。

对游戏来说，这个闭环通常是：

```text
目标 -> 操作 -> 反馈 -> 再尝试
```

对学习工具来说，这个闭环通常是：

```text
出题/情境 -> 回忆/作答 -> 即时反馈 -> 针对薄弱点重复
```

如果这个闭环不能成立，周边系统再完整也无法拯救产品。

---

## 2. 为什么不能先做完整系统

早期很容易把产品设计成一个“大而全”的功能地图：

- 完整关卡系统
- 完整题库系统
- 成就系统
- 用户进度系统
- 移动端打包
- 可配置扩展
- 长期成长路线

这些内容可能都是对的，但它们回答的是：

> 这个产品最终可以长成什么样？

而早期真正要回答的是：

> 用户用 5 分钟之后，是否真的更接近他的目标？

如果这个问题没被验证，系统设计越完整，越可能把错误方向固化。

---

## 3. 方法步骤

### 3.1 找到真实痛点

不要从“我要做哪些功能”开始，而要从用户当前卡在哪里开始。

问题模板：

- 用户现在能做到什么？
- 用户现在做不到什么？
- 用户卡住时的具体场景是什么？
- 用户希望这个能力变成什么样？
- 这个能力能否在 3-7 天内被观察到变化？

Guitar Lab 示例：

```text
用户能看六线谱并弹简单歌曲，
但主要依赖手部肌肉记忆，
无法立即反应音名和 G 大调唱名，
导致能弹但不能顺畅唱出旋律。
```

### 3.2 定义最小能力变化

把“我想学吉他”收缩为一个可训练、可观察的变化。

Guitar Lab 的第一阶段能力变化：

```text
看到六线谱/指板位置
  -> 更快说出音名
  -> 更快说出当前调唱名
  -> 更容易唱出旋律
```

这比“做一个吉他学习 App”更可执行。

### 3.3 设计最小闭环

最小闭环必须包含四件事：

| 环节 | 作用 |
|------|------|
| 输入情境 | 触发用户要练的真实能力 |
| 用户动作 | 用户做出可判断的反应 |
| 即时反馈 | 让用户知道正确映射 |
| 重复机制 | 让薄弱点再次出现 |

Guitar Lab MVP：

```text
高亮指板/显示六线谱
  -> 用户回答音名或唱名
  -> 立即显示位置、音名、唱名
  -> 答错/反应慢/F# 相关题目提高出现权重
```

### 3.4 明确不做清单

早期最需要保护的不是功能列表，而是注意力。

不做清单的价值：

- 防止范围扩散
- 防止架构先行
- 防止把“长期正确”误当成“现在必要”

Guitar Lab 第一版不做：

- 音程几何
- 和弦内音
- CAGED
- 节拍器
- 节奏打分
- 复杂关卡地图
- 移动端打包

这些不是被否定，而是暂时不参与验证核心闭环。

### 3.5 用短周期验证

最小闭环做出来后，用短周期验证。

建议周期：

```text
3-7 天
```

验证指标应该接近真实目标，而不是只看功能是否完成。

Guitar Lab 指标：

- G 大调音名反应时间是否降低
- G 大调唱名反应时间是否降低
- F# 相关题目是否变熟
- 用户弹熟悉旋律时是否更容易唱出唱名

---

## 4. 与游戏开发的对应关系

游戏开发里常说：

> 先做核心玩法，再做外围系统。

学习工具中的对应关系是：

| 游戏开发 | 学习工具 |
|----------|----------|
| 核心玩法 | 最小训练闭环 |
| 手感 | 练习反馈速度与清晰度 |
| 关卡 | 训练情境 |
| 难度曲线 | 认知负荷曲线 |
| 重玩欲望 | 每日练习意愿 |
| 数值系统 | 进度记录与薄弱点权重 |
| 元系统 | 成就、解锁、统计、社交 |

所以学习产品的早期设计也应该先问：

```text
这个练习本身好不好用？
用户完成一次后，是否愿意再来一次？
用户再来一次时，是否真的更强了一点？
```

---

## 5. 何时再搭建周边系统

周边系统不是不要，而是要等核心闭环被验证后再加。

可以进入周边系统的信号：

- 用户愿意连续使用
- 关键指标有改善
- 用户能清楚说出“它帮我变强在哪里”
- 重复练习开始出现疲劳，需要更多变化
- 数据已经足够支持个性化推荐

这时再加入：

- 关卡地图
- 长期解锁
- 成就系统
- 更复杂的题型
- 自适应出题
- 移动端打包

这些系统才会服务核心体验，而不是遮住核心体验。

---

## 6. Guitar Lab 的应用结论

Guitar Lab 当前的正确第一步不是实现完整 Level 1-6，也不是先把架构做满，而是验证这个闭环：

```text
G/C 大调 0-5 品
  -> 指板/六线谱出题
  -> 回答音名/唱名
  -> 即时绑定位置、音名、唱名
  -> 错题/慢题重复
  -> 7 天观察反应速度是否提升
```

如果这个闭环有效，再扩展到：

1. 反向定位：音名/唱名 -> 指板位置
2. 音程几何
3. 和弦内音
4. 把位音阶
5. 更复杂的练习系统

---

## 7. 简短原则

1. 先验证核心能力变化，再设计完整系统。
2. 先让一次练习有效，再让长期成长漂亮。
3. 先做用户真正卡住的那一步，再做产品看起来完整的部分。
4. 架构服务闭环，不让闭环服务架构。
5. 周边系统只有在核心闭环成立后才有价值。
