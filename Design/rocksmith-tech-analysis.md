## 2. 游戏引擎：Gamebryo

**关键发现：Rocksmith 使用的是 Gamebryo 引擎，而非 Unity 或 Unreal。**

### Gamebryo 引擎简介

Gamebryo 是一个老牌 C++ 游戏引擎，特点：

| 特性 | 说明 |
|------|------|
| **语言** | C++（核心），脚本支持 Lua/Python |
| **架构** | 模块化、可扩展的组件系统 |
| **知名作品** | 《上古卷轴4：湮灭》《辐射3》《Catherine》《Rocksmith》 |
| **渲染** | 自己的渲染管线（非基于 Unity/UE） |
| **跨平台** | Windows / Xbox 360 / PS3 / Wii |

### 为什么育碧选 Gamebryo？

1. **2011年的时间点**：Unity 还未成熟到能处理这种级别的音频 DSP，UE3 偏重3A大作
2. **音频集成需求**：Gamebryo 的模块化架构允许深度定制音频处理管线
3. **主机平台支持**：当时 Gamebryo 在 Xbox 360 / PS3 上有成熟的发布经验
4. **自研能力**：育碧作为大厂，有能力基于 Gamebryo 做深度定制

**对我们的启示**：
- Rocksmith 的音频检测引擎是**自研的 C++ DSP 模块**，不是现成的第三方库
- 这解释了为什么它的音高检测能做到消费级产品中的顶尖水平——核心算法是定制的

## 3. 硬件层：Real Tone Cable

### 3.1 硬件规格

```
吉他（1/4"输出） → Real Tone Cable → USB → PC/主机
                          ↓
                    内置 Hercules 芯片
                    （模拟→数字转换）
```

| 参数 | 规格 |
|------|------|
| **ADC 芯片** | Hercules 音频处理芯片 |
| **采样率** | 48 kHz |
| **位深** | 16-bit |
| **频率范围** | 优化覆盖 80Hz ~ 5kHz（吉他有效频段） |
| **延迟** | ~11ms（输入到检测） |
| **信号类型** | 原始模拟波形（非 MIDI！） |

### 3.2 为什么不用 MIDI？

| 对比项 | Real Tone Cable（音频波形） | MIDI 转换器 |
|--------|---------------------------|------------|
| **拾取的信息** | 完整波形（基频+泛音+瞬态+动态） | 仅 note-on/note-off + 力度 |
| **滑弦（Slide）** | ✅ 可检测音高连续变化 | ❌ 无法检测 |
| **推弦（Bend）** | ✅ 可检测音高偏移 | ❌ 无法检测 |
| **闷音（Palm Mute）** | ✅ 通过波形特征识别 | ❌ 无法区分 |
| **泛音（Harmonics）** | ✅ 频谱分析可识别 | ❌ 丢失 |
| **揉弦（Vibrato）** | ✅ 音高周期性波动 | ❌ 丢失 |

**核心洞察**：Rocksmith 的音高检测必须基于**原始音频波形分析**，而非 MIDI 事件。这意味着它需要：
1. 实时 FFT 频谱分析
2. 基频提取（Fundamental Frequency Estimation）
3. 泛音列分析
4. 时域特征（起音、衰减）分析

### 3.3 Rocksmith+ 的扩展：手机麦克风模式

Rocksmith+ 支持通过 **手机麦克风** 采集原声吉他/电吉他声音：
- 使用设备内置麦克风或外接麦克风
- 通过 **Rocksmith Tuner App** 作为蓝牙音频桥接
- 检测精度相比 Real Tone Cable 有所下降，但降低了门槛

## 4. 音频检测核心技术（推测分析）

> 注意：育碧未公开 Rocksmith 的核心 DSP 算法细节。以下基于公开信息、DSP 学术文献和社区逆向工程的推测。

### 4.1 整体信号处理流程

```
吉他拾音器 → Real Tone Cable ADC → USB 音频流
                                            ↓
                                    ┌──────────────────┐
                                    │  预处理阶段       │
                                    │  - 降噪/滤波      │
                                    │  - 增益标准化     │
                                    │  - 噪声门限       │
                                    └────────┬─────────┘
                                             ↓
                                    ┌──────────────────┐
                                    │  特征提取阶段     │
                                    │  - FFT 频谱分析   │
                                    │  - 基频检测       │
                                    │  - 泛音列分析     │
                                    │  - 起音检测       │
                                    └────────┬─────────┘
                                             ↓
                                    ┌──────────────────┐
                                    │  匹配阶段         │
                                    │  - 与谱面音符对比 │
                                    │  - 音高容差判断   │
                                    │  - 时序容差判断   │
                                    └────────┬─────────┘
                                             ↓
                                    游戏反馈（命中/偏差/错过）
```

### 4.2 多音检测（Polyphonic Pitch Detection）

Rocksmith 面临的最难问题：**同时检测多根弦的发音**。

当你在吉他上扫一个和弦时，六根弦可能同时响，信号是多个正弦波的叠加。Rocksmith 需要从中分离出每个音的音高。

#### 可能的算法路径

**路径A：频域主导分析（最可能）**

```
1. 对音频帧做加窗 FFT（如 Blackman-Harris 窗）
2. 在频谱中寻找基频峰值
3. 通过泛音列关系验证基频（基频的整数倍处应有能量峰值）
4. 迭代检测：找到最强基频 → 消除其泛音影响 → 找下一个基频
5. 对每个检测到的音高，匹配到谱面中的预期音符
```

**路径B：时频联合分析**

```
1. 短时傅里叶变换（STFT）得到时频图
2. 在每个时间点，分析频率峰值
3. 使用动态规划追踪音高轨迹（类似语音中的 pitch tracking）
4. 结合 onset 检测确定音符起始时刻
```

**路径C：机器学习（Rocksmith+ 可能采用）**

```
1. 将频谱图输入 CNN 或 RNN
2. 网络输出多音高概率分布
3. 后处理得到离散的音符事件
```

### 4.3 社区逆向工程的发现

Rocksmith 社区（Customsforge）通过分析游戏文件和音频行为，发现了一些特征：

| 发现 | 说明 |
|------|------|
| **固定采样率** | 游戏强制使用 48kHz/16-bit，即使你的声卡支持更高 |
| **Exclusive Mode** | Windows 版支持独占音频模式，减少系统混音带来的延迟 |
| **Latency Buffer** | 可调缓冲区大小（1-4），影响延迟与稳定性权衡 |
| **Win32UltraLowLatencyMode** | 配置项，启用 Windows 的极低延迟音频路径 |
| **检测宽容度** | 音高有一定容差（约 ±10-20 音分），不是绝对精确 |
| **时序窗口** | 音符必须在时间窗口内演奏，过早/过晚都算错过 |

### 4.4 与 PolyTune（多音调音器）的对比

PolyTune（TC Electronic 的多音吉他调音器）和 Rocksmith 面临相似的技术挑战：

| 产品 | 应用场景 | 检测方式 |
|------|---------|---------|
| **PolyTune** | 调音（静态和弦） | 频域峰值检测 + 谐波分析 |
| **Rocksmith** | 实时演奏（动态音符序列） | 时频联合分析 + 音高追踪 + 时序匹配 |

PolyTune 只需告诉你"当前六根弦分别是什么音"，而 Rocksmith 还要：
- 追踪音高随时间的变化（推弦、滑弦）
- 与谱面中的预期音符实时匹配
- 判断演奏时序是否准确

## 5. 延迟（Latency）分析

延迟是吉他游戏的生死线。如果玩家拨弦后过很久才听到反馈，体验会极差。

### 5.1 Rocksmith 的延迟构成

```
总延迟 = ADC转换延迟 + USB传输延迟 + 音频缓冲区延迟 + DSP处理延迟 + 游戏逻辑延迟 + DAC输出延迟
         ↑ Real Tone    ↑ USB协议    ↑ 可配置          ↑ 音高检测      ↑ 命中判断      ↑ 耳机/音箱
           Cable                                        算法
```

### 5.2 实测数据（社区测试）

| 配置 | 总延迟 |
|------|--------|
| Real Tone Cable + 默认设置 | ~30-40ms |
| Real Tone Cable + 优化设置（LatencyBuffer=1） | ~20-25ms |
| 专业声卡（Focusrite Scarlett）+ RS_ASIO | ~15-20ms |
| 手机麦克风模式（Rocksmith+） | ~50-100ms+ |

### 5.3 延迟优化策略

Rocksmith 采用的优化手段：

1. **独占音频模式**：绕过 Windows 系统混音器，直接访问声卡
2. **小缓冲区**：可配置 128/256/512 sample 的缓冲区
3. **音频线程优先级**：将音频处理线程设为高优先级
4. **预测性检测**：可能采用轻微的预测算法，提前判断音符命中

## 6. 谱面数据格式

Rocksmith 的"音符高速公路"需要精确的谱面数据：

```
歌曲音频文件（多轨：吉他、贝斯、鼓、人声、伴奏）
    ↓
谱面文件（.psarc 格式，加密压缩）
    ├── 音符事件：时间戳 + 弦号 + 品号 + 技巧（推弦/滑弦/泛音等）
    ├── 动态难度：同一首歌有多个难度层级
    ├── 段落标记：前奏/主歌/副歌/间奏/Solo
    └── 音色配置：每个段落使用的效果器/音箱模拟参数
```

### 6.1 动态难度系统（Dynamic Difficulty）

Rocksmith 的核心创新之一：

```
玩家表现评估
    ↓
实时调整谱面难度
    ├── 表现好 → 逐步增加音符密度
    ├── 表现差 → 减少音符，只保留关键音符
    └── 自适应：不同段落可以有不同的难度级别
```

这与我们设计中提到的"难度曲线"理念一致。

## 7. 音频渲染：吉他效果器/音箱模拟

Rocksmith 不仅是检测设备，还需要让吉他声音好听：

```
吉他输入信号
    ↓
数字信号处理（DSP 效果器链）
    ├── 噪声门（Noise Gate）
    ├── 压缩器（Compressor）
    ├── 失真/过载（Distortion/Overdrive）
    ├── EQ
    ├── 调制效果（合唱/镶边/相位）
    ├── 延迟/混响（Delay/Reverb）
    └── 音箱模拟（Cabinet Simulation）
    ↓
输出到耳机/音箱
```

这些效果器链是**预配置**的，根据歌曲风格自动切换。玩家也可以自定义音色。

## 8. 移动端 Rocksmith+ 的技术挑战

Rocksmith+ 登陆 iOS/Android 面临新的技术问题：

| 挑战 | 解决方案（推测） |
|------|----------------|
| **低质量麦克风输入** | 手机麦克风频响不平坦，需要频谱补偿算法 |
| **环境噪声** | 降噪算法 + 信噪比阈值判断 |
| **高延迟** | 蓝牙音频桥接延迟大，可能采用本地处理 |
| **CPU 限制** | 移动设备算力有限，可能简化 DSP 算法 |
| **电池消耗** | 持续音频处理耗电，需要优化算法效率 |

## 9. 对我们项目的启示

### 9.1 技术层面

| Rocksmith 的做法 | 我们可借鉴的 |
|-----------------|-------------|
| **原始波形分析而非 MIDI** | 我们的节奏打分也应基于原始音频，而非依赖 MIDI 输入 |
| **FFT + 频谱分析** | Web Audio API 的 AnalyserNode 可以做同样的频谱分析 |
| **泛音列验证基频** | 吉他音高检测中，通过泛音关系验证可提高准确性 |
| **时序窗口容差** | 节奏打分中设置合理的命中窗口（如 ±50ms） |
| **动态难度** | 我们的指板游戏也可以根据用户表现自适应调整 |

### 9.2 产品层面

| Rocksmith 的经验 | 我们的差异化 |
|-----------------|-------------|
| **硬件门槛**（需要 Real Tone Cable） | 我们只用手机麦克风，零硬件门槛 |
| **曲库授权成本高** | 我们不涉及歌曲版权，专注基本功训练 |
| **3A级开发投入** | 我们轻量快速验证，MVP 优先 |
| **主机/PC 为主** | 我们网页版优先，移动端 App 跟进 |

### 9.3 技术可行性评估

| Rocksmith 功能 | 我们的实现路径 | 难度 |
|---------------|---------------|------|
| 单音音高检测 | Web Audio API + 自相关/FFT | ⭐⭐ 中等 |
| 多音（和弦）检测 | FFT + 峰值分离 | ⭐⭐⭐ 较难 |
| 推弦/滑弦检测 | 音高追踪 + 变化率分析 | ⭐⭐⭐ 较难 |
| 节奏对齐检测 | Onset Detection + 时间戳对比 | ⭐⭐ 中等 |
| 吉他效果器渲染 | Tone.js 效果器链 | ⭐⭐ 中等 |
| 动态难度 | 基于正确率的算法调整 | ⭐⭐ 中等 |

## 附录：Rocksmith 配置文件的音频参数

```ini
; Rocksmith.ini 音频配置示例
[Audio]
EnableMicrophone=0
ExclusiveMode=1
LatencyBuffer=2
ForceDefaultPlaybackDevice=0
ForceWDM=0
ForceDirectXSink=0
DumpAudioLog=0
MaxOutputBufferSize=0
RealToneCableOnly=0
Win32UltraLowLatencyMode=1
```

| 参数 | 说明 |
|------|------|
| `ExclusiveMode` | 独占音频模式，减少系统混音延迟 |
| `LatencyBuffer` | 缓冲区大小级别（1=最小/最快，4=最大/最稳定） |
| `Win32UltraLowLatencyMode` | Windows 极低延迟模式 |
| `RealToneCableOnly` | 是否只允许 Real Tone Cable |
| `DumpAudioLog` | 输出音频调试日志 |
